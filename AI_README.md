# AI Blog Writing Guide

本文件用于约束 agent/AI 工具在本仓库中撰写、修改博客文章时的行为。

目标只有一个：让 AI 产出的文章能直接放进当前 Hexo 博客，不需要再大幅返工。

## 适用范围

- 仅适用于本仓库博客内容写作
- 主要作用目录：
  - `source/_drafts/`
  - `source/_posts/`
  - `source/about/`
  - `source/images/posts/`
- 不适用于主题开发、依赖升级、部署脚本改造

## 总原则

- 一律使用中文写作，除标签名、算法名、库名、平台名外
- 内容必须技术准确，不能为了凑字数写空话
- 优先贴合本仓库已有文章风格，而不是自由发挥
- 非必要不创造新目录、新分类体系、新命名风格
- 不允许引用会过期的外链图片
- 不允许把密钥、token、账号信息、私密链接写进仓库

## 文件位置

- AI 新写文章必须先放在 `source/_drafts/`
- 只有人工确认完成后，草稿才可以转移到 `source/_posts/`
- `source/_posts/` 视为正式发布内容目录，AI 不应直接写入
- 文章内使用的封面图、插图优先放在 `source/images/posts/`
- 废弃但暂不删除的图片放在 `source/images/trash/`
- 不要再使用 `*.assets/` 目录保存正文图片，除非用户明确要求

推荐草稿路径示例：

- `source/_drafts/2025-03-21-BFS-flood-fill.md`
- `source/_drafts/mst-note.md`

## Front Matter 规范

每篇文章必须有 YAML front matter，推荐字段顺序如下：

```yaml
---
categories:
- 一级分类
- 二级分类
cover: /images/posts/xxx-cover.svg
date: 2025-03-21 02:42:08
index_img: /images/posts/xxx-cover.svg
tags:
- cpp
- algorithm
- ACwing
- BFS
title: 2025-03-21文章标题
---
```

要求如下：

- `categories` 必填，使用数组形式
- `tags` 必填，使用数组形式
- `cover` 和 `index_img` 默认都要填写，且保持一致
- `date` 使用 `YYYY-MM-DD HH:mm:ss`
- `title` 必填，不要留模板值
- front matter 只保留必要字段，不要随意塞自定义键

## 标题规范

- 默认采用 `YYYY-MM-DD标题` 或 `YYYY-MM-DD[专题]标题`
- 如果是某个算法专题，允许使用：
  - `2025-03-21[BFS]Flood-Fill&最短路径`
  - `2025-02-06最小生成树的典型应用`
- 不要写成营销标题、口语标题、问句标题
- 不要故意制造夸张表达，如“保姆级”“一文学会”“史上最全”

## 正文标题层级

正文结构默认使用以下层级：

```markdown
# 文章标题
## 目的 / 主要内容 / 核心思想
### 具体知识点或例题名
```

要求如下：

- `#` 标题必须与文章主题一致
- 不要跳级，避免 `#` 后直接接 `###`
- 标题命名要直白，优先使用“定义 / 思路 / 模板 / 例题 / 注意点”这类词

## 分类规范

优先复用现有分类体系，不随意发明新的一级分类。

当前仓库常见分类风格：

- `寒假练题计划 / 基础算法`
- `寒假练题计划 / 提高算法 / 搜索`
- `寒假练题计划 / 提高算法 / 图论 / 最小生成树`
- `博客搭建`
- `实用工具推荐`

要求如下：

- 算法题解优先挂到已有算法分类下
- 非算法文章不要硬塞到算法分类
- 除非用户明确要求，不新增一级分类

## 标签规范

标签使用已有风格，尽量简洁稳定。

常见标签风格：

- `cpp`
- `algorithm`
- `ACwing`
- `BFS`
- `DFS`
- `MST`
- `Kruskal`
- `Prim`
- `Disjoint-set`
- `Shortest-Path`
- `big integer`

要求如下：

- 标签数量建议 3 到 8 个
- 平台标签、语言标签、算法标签可以同时存在
- 不要添加情绪化标签、口语化标签、无信息量标签
- 不要重复同义标签

## 图片规范

- 封面图默认使用本地静态资源
- 封面路径统一写成 `/images/posts/...`
- 正文插图也优先使用 `/images/posts/...`
- 不允许直接使用会失效的外链图床
- 不允许引用需要登录、会过期、会防盗链的图片地址

封面建议命名：

- `topic-cover.svg`
- `topic-cover.png`
- `topic-cover.jpg`

例如：

- `/images/posts/bfs-flood-fill-cover.svg`
- `/images/posts/mst-cover.svg`

如果 AI 无法准备合适封面：

- 可以先复用现有稳定封面
- 或生成简单的本地静态 `svg` 封面
- 但不要留空，不要继续引用外链

## 内容风格规范

- 以“笔记 + 题解 + 模板总结”为主
- 解释必须围绕算法本身，不写泛泛而谈的背景故事
- 先讲概念，再讲思路，再讲代码或例题
- 重点说明：
  - 为什么这样做
  - 时间复杂度/空间复杂度
  - 易错点
  - 适用场景

禁止：

- 过度口语化
- 过度营销化
- 大段无意义套话
- 没有验证就编造复杂度、结论或题意

## 代码规范

默认贴完整、可读、与仓库风格一致的 C++ 代码。

遵循当前仓库已有风格：

- 默认使用 `cpp` 代码块
- 常见写法可保持：
  - `#include<bits/stdc++.h>`
  - `using namespace std;`
- 变量命名、代码排版尽量和现有文章接近
- 注释以简洁中文为主

要求如下：

- 代码块必须带语言标识：

```cpp
// code
```

- 如果来源于题目，优先在代码顶部注明题目链接
- 如果写模板，说明模板解决的问题
- 不要只给残缺代码，除非用户明确要求只要片段

## 例题部分规范

如果文章是算法笔记，优先包含“例题”部分。

推荐结构：

```markdown
## 例题
### 题目名
```cpp
// 题目链接
```
```

要求如下：

- 例题名要明确
- 例题和上文知识点要能对应起来
- 不要堆很多题名却没有分析价值

## 写作流程要求

当 agent 新写一篇文章时，默认流程应为：

1. 确定文章主题
2. 选择最贴近的现有分类
3. 确定标签
4. 生成 front matter
5. 按既有风格写正文
6. 补本地封面或选择稳定本地封面
7. 检查 Markdown 渲染是否自然
8. 将文章保存到 `source/_drafts/`
9. 确认没有外链敏感资源、没有密钥、没有无效图片
10. 等待人工审核，再决定是否迁移到 `source/_posts/`

## 修改已有文章时的要求

- 优先做增量修改，不要整篇重写
- 保留原作者已有风格和有效内容
- 除非用户明确要求，不要擅自改标题、分类、日期
- 如果只修图片或 front matter，不要顺手大改正文
- 如果目标是“新写一篇”，默认创建草稿而不是正式文章

## 明确禁止事项

- 不要修改 `.deploy_git/` 内容
- 不要修改 `public/` 里的生成产物来“伪修复”
- 不要把 `.env`、token、PAT、账号密码写进仓库
- 不要把外链 HTML 页面误保存成图片文件
- 不要创建无意义临时文件
- 不要为了“美观”擅自改整站配置
- 不要未经确认直接把 AI 文章写入 `source/_posts/`

## 文章完成前自检清单

在提交文章前，agent 必须自查：

- front matter 是否完整
- `cover` 与 `index_img` 是否存在且有效
- 图片是否全部本地静态化
- 分类与标签是否合理
- 标题格式是否符合仓库风格
- Markdown 标题层级是否正常
- 代码块是否标注语言
- 是否存在明显事实错误、错别字、占位文本

## 最终目标

AI 生成的博客文章应满足：

- 直接可合入仓库
- 风格与现有文章一致
- 技术表达清楚
- 图片、链接、配置都稳定
- 不引入额外维护负担
